# App 3：AI 校園心靈守護者 ── 我的程式邏輯

---

## 作品動機

我在學校有時候會看到同學悶悶不樂，但老師很忙，不一定能及時注意到每一個學生。
我在想：**如果有一個系統，能自動偵測到學生可能情緒不好，提醒老師去關心，會不會讓校園更溫暖？**

所以我設計了「AI 校園心靈守護者」——它不是要取代老師，而是幫老師多一雙眼睛。
偵測到異常訊號後，AI 會分析風險等級，給出具體建議，讓老師知道**該怎麼關心、關心什麼**。

為了保護學生隱私，系統不記錄真實姓名，只用「小明」「小花」等別名。
這是我覺得這個作品最重要的設計原則：**幫助人，但不傷害人。**

---

## 系統架構（三層）

```
感測層（硬體）
    ↓ 異常數值 → Firebase 即時資料庫
AI 分析層（雲端）
    ↓ 風險等級 + 關懷建議 → App 介面
行動層（老師 + 硬體）
    ↓ 老師操作 → CARE_DEPLOYED 指令 → Arduino R4 WiFi
```

### 感測層

硬體使用 **Arduino UNO R4 WiFi**，透過 Wi-Fi 連接 Firebase。
感測節點裝在校園各個位置（教室、走廊、操場），持續回傳環境數值。
一旦偵測到異常，就把資料寫入 Firebase 即時資料庫。

### AI 分析層

後端使用 **Gemini AI API**，負責：
1. 讀取感測資料
2. 判斷風險等級（高 / 中 / 低）
3. 生成具體的關懷建議文字
4. 把結果存回 Firebase，讓前端即時顯示

### 行動層

前端是用 **React** 做的 App，老師打開後可以：
- 看到即時警示
- 閱讀 AI 建議
- 勾選關懷清單
- 按「部署關懷」→ 系統發出 `CARE_DEPLOYED` 指令給 Arduino，啟動對應動作

## 主要邏輯

### 持續監控

```
程式啟動：
    建立 Firebase 即時資料庫的監聽器
    重複無限次：
        等待感測節點傳來新數值
        把新數值更新到 App 的校園地圖上
        如果 數值有異常：
            呼叫「風險評估」函式
        結束如果
    結束重複
```

> **說明**：Firebase 的「即時資料庫」有個很酷的功能叫做「監聽器」（listener）。
> 不需要一直問「有沒有新資料」，有資料自動推過來，就像訂閱通知一樣。
> 這讓 App 不會浪費電、不會有延遲。

---

### 風險評估（用 if-else 架構）

```
函式 風險評估（感測數值、節點位置、別名）：

    把資料送給 Gemini AI 分析

    如果 AI 回傳風險等級 == "高"：
        觸發高風險警示（紅色）
        發出 ALERT_SIGNAL 指令給 Arduino
        在校園地圖上把該節點標紅
        建立新警示記錄，狀態 = "新提醒"
        立即推播通知給老師

    否則 如果 AI 回傳風險等級 == "中"：
        觸發中風險警示（橘色）
        在校園地圖上把該節點標橘
        建立新警示記錄，狀態 = "新提醒"
        推播通知給老師

    否則 如果 AI 回傳風險等級 == "低"：
        觸發低風險提示（黃色）
        在校園地圖上把該節點標黃
        記錄到系統，狀態 = "觀察中"
        不立即推播，在 App 內顯示提示即可

    否則：
        -- 正常範圍，不做任何事
        繼續監控

    結束如果

    把 AI 生成的關懷建議文字存回 Firebase
    回傳 警示記錄編號
```

> **我學到的重點**：AI 不是魔法，它其實就是一個「很複雜的 if-else」。
> 我給 Gemini AI 一個問題（「這個數值代表什麼？」），它根據訓練好的模型給我答案。
> 我的程式再根據答案決定要做什麼動作。**AI 分析 + 程式決策，缺一不可。**

### 關懷部署流程

```
老師操作流程：

    步驟 1：老師收到推播通知
    步驟 2：老師打開 App，查看警示詳情
        → 顯示：別名（例：小明）、節點位置（走廊B區）、時間、AI 建議
    步驟 3：老師逐項勾選關懷清單
        例：
        [✓] 主動走近學生詢問
        [✓] 詢問睡眠與飲食狀況
        [ ] 聯絡家長
        [ ] 轉介輔導室

    步驟 4：如果 所有清單項目都勾選了：
        啟用「部署關懷」按鈕
    結束如果

    步驟 5：老師按下「部署關懷」
        → 把警示狀態改成 "處理中"
        → 發出 CARE_DEPLOYED 指令給 Arduino（Arduino 會啟動回應動作）
        → Firebase 記錄操作時間與老師代號

    步驟 6：老師確認完成後，按「標記結案」
        → 把警示狀態改成 "已結案"
        → 感測節點恢復「正常」標示
        → 這筆記錄進入歷史案件清單（可供日後查閱）
```

> **說明**：我設計成「清單全勾才能按部署」，是因為我希望老師真的去思考每個關懷步驟，
> 不是隨便點一下就結案。這是**人性化設計**，確保技術真的有幫助到人。

---

### 隱私保護機制

```
隱私設計原則：

    -- 1. 別名系統
    每個學生在系統中只有一個隨機別名（小明、小花、小智…）
    真實姓名不會出現在 App 介面、不儲存在 Firebase
    老師在操作時只看到別名，對應關係只存在老師的實體記錄中

    -- 2. 感測資料最小化
    只記錄「有沒有異常」，不記錄詳細的個人生理數值
    資料在分析完後，原始數值會被刪除，只留下風險等級結論

    -- 3. 資料存取控制（Firebase 安全規則）
    只有登入系統的老師帳號才能讀取警示記錄
    學生無法查看自己或別人的記錄

    -- 4. 已結案資料保留期限
    超過 90 天的已結案記錄，系統自動提醒是否刪除
```

> **說明**：我覺得做這個系統最難的部分不是寫程式，而是「如何在幫助學生的同時，
> 不讓學生覺得被監視」。隱私保護是我在這個作品裡花最多時間思考的設計問題。

## 我學到的技術概念

### 1. Firebase 即時資料庫

Firebase 是 Google 提供的雲端資料庫服務。它特別的地方是「即時同步」：
資料一更新，所有連線的裝置（手機、電腦）馬上就會看到新內容，不需要重新整理頁面。
我用它來存放感測節點的狀態、警示記錄、關懷清單進度。

### 2. AI 推理邏輯

我以前以為 AI 是「自己想出答案」，但其實 AI 是根據大量訓練資料學習到「什麼輸入對應什麼輸出」。
我使用 Gemini AI API，把感測數值和情境描述傳進去，AI 回傳分析結果。
**我的程式控制「要問什麼」和「拿到答案後做什麼」，AI 負責「分析判斷」。**

### 3. 隱私保護設計

不是只有程式要寫得安全，**資料的收集方式本身就要有隱私的思維**。
我學到「資料最小化原則」——只收集絕對必要的資料，減少洩漏的風險。
這不只是技術問題，也是設計倫理問題。

### 4. 硬體與軟體整合

Arduino R4 WiFi 負責硬體端的動作，React App 負責使用者介面。
兩者之間透過序列指令（Serial Commands）溝通：App 送出文字指令，Arduino 接收後執行對應動作。
例如 `ALERT_SIGNAL` → Arduino 控制警示燈閃爍；`CARE_DEPLOYED` → 觸發其他機構動作。

### 5. 狀態機設計

一個警示從「新提醒」→「處理中」→「已結案」，每個狀態有明確的進入條件和轉換規則。
這種「狀態機」的思維讓我的程式邏輯更清晰，不會出現「不知道現在到底在哪個步驟」的混亂。

---

## 作品總結

這個作品結合了**感測硬體、雲端資料庫、AI 分析、隱私保護**四個技術領域，
但我覺得最核心的不是技術，而是設計理念：

> **用科技幫助老師，讓每個學生都能被及時看見。**

程式是工具，善用工具才是重點。
